# NCC Section Compliance Checker - Update Plan

## 1. Project Overview
- Create a new backend structure (backend2) while maintaining existing functionality
- Implement a more flexible and maintainable report generation system
- Keep existing valuable mappings and data

## 2. New Backend Structure
```
backend2/
├── config/
│   ├── db.config.js         # Database configuration
│   └── env.config.js        # Environment configuration
├── models/
│   ├── project.model.js     # Project information
│   ├── user.model.js        # User authentication
│   ├── climateZone.model.js # Climate zone data
│   └── requirement.model.js # NCC requirements and conditions
├── controllers/
│   ├── project.controller.js # Project CRUD operations
│   ├── user.controller.js    # User authentication
│   └── report.controller.js  # Report generation
├── services/
│   ├── report.service.js     # Report generation logic
│   └── calculation.service.js # Calculation utilities
├── utils/
│   ├── decisionTree.utils.js # Decision tree processing
│   ├── location.utils.js     # Location utilities
│   └── validation.utils.js   # Data validation
├── data/
│   ├── decision-trees/      # New decision tree structure
│   │   ├── report-structure.json
│   │   ├── j1p2-calculations.json
│   │   └── ... (other topic files)
│   └── mappings/            # Existing mappings
│       ├── locationToClimateZone.json
│       └── buildingTypeToClassification.json
├── routes/
│   ├── project.routes.js    # Project routes
│   ├── user.routes.js       # User routes
│   └── report.routes.js     # Report routes
├── middleware/
│   ├── auth.middleware.js   # Authentication
│   └── validation.middleware.js # Request validation
└── app.js                   # Main application file
```

## 4. Data Structure Details

### Report Structure Definition
```json
{
  "version": "1.0",
  "sections": [
    {
      "id": "general",
      "title": "General",
      "order": 1,
      "topics": [
        {
          "id": "project_info",
          "file": "project-info.json",
          "order": 1
        }
        // ... other topics
      ]
    }
    // ... other sections
  ]
}
```

### Topic File Structure
```json
{
  "version": "1.0",
  "base_conditions": {
    "building_class": ["Class_2", "Class_4"],
    "climate_zone": "0",
    "total_area": {
      "min": 0,
      "max": 10000
    },
    "total_area_habitable_rooms": "0"
  },
  "sections": [
    {
      "id": "section_1",
      "title": "Section Title",
      "order": 1,
      "conditions": {
        "building_class": ["Class_2"],
        "climate_zone": ["1", "2", "3"],
        "total_area": {
          "min": 50,
          "max": 200
        }
      },
      "description": "Section description",
      "content": {
        // Section-specific content
      }
    }
    // ... more sections
  ]
}
```

## 6. Migration Strategy

1. **Phase 1: Parallel Development**
   - Develop new backend2 while maintaining existing backend
   - Share database between both systems
   - Implement new features in backend2

2. **Phase 2: Feature Migration**
   - Migrate features one by one to backend2
   - Test each migrated feature
   - Maintain backward compatibility

3. **Phase 3: Full Migration**
   - Complete migration of all features
   - Update frontend to use new backend
   - Decommission old backend
